## What this app does

STT Audio Segmentor is a desktop GUI app (built with Wails v2) that slices a WAV or M4A audio file into segments based on timestamps read from an Excel or CSV spreadsheet. It outputs individual `.wav` files plus a `transcripts.txt` in a `<basename>_segments/` folder next to the source audio. M4A files are first converted to WAV via `ffmpeg`.

## Commands

### Development
```bash
wails dev          # Run in dev mode (hot-reloads frontend, live-reloads Go on save)
```

### Build
```bash
wails build        # Produce a native binary in build/bin/
```

### Frontend only (rarely needed standalone)
```bash
cd frontend && npm run dev      # Vite dev server
cd frontend && npm run build    # Build frontend assets into frontend/dist/
```

### Go
```bash
go build ./...     # Compile check
go test ./...      # Run tests
```

**Prerequisite:** `ffmpeg` must be installed (`brew install ffmpeg` on Mac) for M4A support.

## Architecture

This is a two-layer Wails v2 app:

- **Go backend (`app.go`, `main.go`)** — all business logic lives here. The `App` struct is bound to the Wails runtime and its exported methods become callable from the frontend via auto-generated bindings.
  - `Run(excelPath, audioPath, targetCol, targetText)` — main entry point; reads the spreadsheet, optionally converts M4A→WAV via ffmpeg, decodes the WAV with `go-audio/wav`, slices it, writes segment files, and emits `"progress"` events back to the frontend.
  - `SelectFile(...)` — wraps Wails' native file-picker dialog.
  - `ReadFileAsBase64(path)` — reads a file from disk and returns base64; used by the frontend to load audio for in-browser playback without a file:// URL.

- **React frontend (`frontend/src/App.jsx`)** — calls Go methods via `wailsjs/go/main/App` auto-generated bindings and listens for `"progress"` events via `wailsjs/runtime`. No build step needed during `wails dev`; Wails proxies the Vite dev server.

- **Wails bindings** (`frontend/wailsjs/`) — auto-generated by Wails from the Go `App` struct. Do not edit manually; they regenerate on `wails dev`/`wails build`.

## Key data flow

1. User picks a spreadsheet and audio file in the UI.
2. Frontend calls `Run(...)` (Go).
3. Go reads timestamps and texts from the spreadsheet (CSV or Excel, first sheet, column names configurable — defaults: `"Relative Timestamp"` and `"Text"`).
4. Go decodes the full WAV into an `audio.IntBuffer`, then slices by sample index for each consecutive timestamp pair.
5. Each segment is written as a new WAV file; the last segment runs to end-of-audio.
6. Progress messages stream back to the frontend via Wails events (`"progress"`).
7. On completion, `ProcessResult` (success, message, segment list) is returned to the frontend, which renders a playable segment list using in-memory blob URLs.

## Timestamp format

Timestamps in the spreadsheet must be `MM:SS` or `HH:MM:SS`. Rows with empty timestamps are skipped silently.
